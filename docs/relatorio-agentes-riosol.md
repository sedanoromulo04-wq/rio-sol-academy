# Estudo dos agentes do RIO SOL Academy

Este documento responde ao conjunto de perguntas que você listou e resume o que o código do repositório já implementa, como os agentes estão encadeados com LLM, memória e controle administrativo, além de apontar o que precisa ser preparado para que tudo funcione em produção.

## 1. Como os agentes foram programados

- O núcleo dos agentes roda no backend Node (`backend/server.mjs`), que expõe três rotas principais: `/api/agents/mentor`, `/api/agents/roleplay` e `/api/agents/mentor-feedback`. Cada chamada passa pelo helper `createAgentReply`, que injeta histórico, memórias de RAG (quando disponíveis), o “extraInstructions” adequado e dispara uma geração via Gemini antes de salvar o par pergunta/resposta no Supabase. Quando o token do usuário está presente, o servidor busca memórias vetoriais com `searchAgentMemories` e, ao final, armazena o novo par usando `storeAgentMemory`. Por trás dos panos, `createAgentReply` monta o prompt com `buildAgentPrompt`, habilita responsabilidades de RAG e retorna referências recuperadas ao frontend junto com o texto da IA. ([backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs))
- O frontend mantém o fluxo vivo em `src/services/ai-mentor.ts`, que encapsula chamadas aos endpoints acima. A tela `src/pages/Simulator.tsx` usa esse serviço para chamar `simulateRoleplay` (atualizado com as mensagens e a persona selecionada), gravar o histórico em `chats` (`saveChatMessage`) e puxar feedback pedagógico via `getMentorFeedback`. Assim, cada simulação alimenta o diálogo e aciona os agentes com contexto do usuário. ([src/services/ai-mentor.ts](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/src/services/ai-mentor.ts), [src/pages/Simulator.tsx](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/src/pages/Simulator.tsx))
- Além do backend Node, existe um integration bridge com o NotebookLM via `backend/notebooklm_bridge.py`. Esse script roda comandos Python (`NotebookLMClient`) para listar cadernos, carregar fontes (texto/URL/arquivo), gerar podcasts e responder perguntas paginadas. A rota `/api/notebooklm/ask` converte as respostas em referências formatadas e retorna `engine: 'notebooklm'`, permitindo usar matérias preparadas nos notebooks do Google. ([backend/notebooklm_bridge.py](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/notebooklm_bridge.py), [src/lib/notebooklm-api.ts](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/src/lib/notebooklm-api.ts))
- Observação: ainda há funções Deno/Supabase (`supabase/functions/simulator-roleplay` e `supabase/functions/zenith-mentor`) que chamam a API da Claude, mas o fluxo atual da UI já aponta para o backend local com Gemini. Isso significa que o código antigo permanece como “fallback” ou referência, mas o prisma do aplicativo de produção passa pelo Node+Gemini+RAG descrito acima. ([supabase/functions/simulator-roleplay/index.ts](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/supabase/functions/simulator-roleplay/index.ts))

## 2. Os agentes têm system prompt?

- Sim. Cada tipo de agente constrói um system prompt dedicado:
  - `buildMentorSystemPrompt` inclui o perfil do usuário, o sumário das cinco últimas atividades, o papel de “mentor comercial sênior” e regras como “sugira frases práticas” e “não dê respostas prolixas”. As instruções podem ser extendidas com os prompts compartilhados e específicos definidos no banco. ([backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs))
  - `buildRoleplaySystemPrompt` entrega a descrição da persona selecionada, estatísticas detalhadas (paciência, conhecimento técnico, sensibilidade financeira) e regras de permanência no personagem (“nunca revele que é IA”, “responda apenas com sua fala”). 
  - `buildMentorFeedbackSystemPrompt` pede um parágrafo direto e pedagógico com sugestão de frase alternativa quando necessário.
- O helper `mergePromptInstruction` junta o prompt base com `promptConfig.shared` e o valor específico (mentor, roleplay ou mentorFeedback). Esse `promptConfig` é carregado via `loadAgentPromptConfig`, que lê a chave `agent_prompt_config` da tabela `system_settings`. Há um fallback (`defaultAgentPromptConfig`) para garantir que os agentes sempre recebam um prompt, mesmo sem overrides. ([backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs), [backend/supabase-rag.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/supabase-rag.mjs))
- Além disso, cada endpoint injeta uma instrução `extraInstructions` fixando comportamentos (“responda como mentor comercial senior” ou “não explique seu raciocínio e não quebre o personagem”) e regula temperatura/max tokens para cada propósito.

## 3. Qual LLM os agentes estão utilizando?

- O backend chama a API do Gemini via `backend/gemini.mjs`. O modelo de resposta padrão é `gemini-2.5-flash` e o embedding padrão é `gemini-embedding-2-preview` com dimensão 768. Os métodos `generateText` e `embedSingleText` disparam POSTs para `https://generativelanguage.googleapis.com/v1beta/models/{model}` usando o `GEMINI_API_KEY` (ou `GOOGLE_AI_API_KEY`). Se a chave estiver faltando, o servidor lança um erro. ([backend/gemini.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/gemini.mjs))
- O endpoint `/health` (e `/api/notebooklm/status`) expõe o resultado de `getGeminiRuntimeConfig`, mostrando se a chave está configurada, qual modelo está em uso e a dimensão dos embeddings. Isso ajuda o time a validar se o serviço está pronto para rodar os agentes. ([backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs"))

## 4. Como o admin pode programar o sistema para ter controle de todos os agentes?

- O painel administrativo (`src/pages/admin/AdminAgents.tsx`) consome a API `agentsAdminApi`, que chama `/api/agents/config`, `/api/agents/memories`, `/api/agents/memories/import`, `/api/agents/memories/clear`, `/api/agents/memories/reindex` e o delete individual. O painel permite editar o prompt global (campo “shared”) e os prompts específicos para mentor, roleplay e mentor-feedback, além de importar textos ou arquivos para a base vetorial e filtrar/excluir memórias existentes por agente, visibilidade ou documento. ([src/pages/admin/AdminAgents.tsx](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/src/pages/admin/AdminAgents.tsx), [src/lib/agents-admin-api.ts](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/src/lib/agents-admin-api.ts))
- O backend exige que o token enviado pelo admin passe por `assertAdminAccess` (que valida `is_admin = true` via a função `public.is_admin()` definida no migration `20260323175000_fix_rls_recursion.sql`). Ou seja, somente perfis com a flag `is_admin` cadastrada na tabela `profiles` (seed adiciona `lucas.salles@riosolenergias.com.br`) conseguem mudar prompts ou manipular memórias. ([supabase/migrations/20260323175000_fix_rls_recursion.sql](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/supabase/migrations/20260323175000_fix_rls_recursion.sql), [backend/supabase-rag.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/supabase-rag.mjs))
- O admin também pode importar material na memória: ao enviar texto ou arquivo, a rota `/api/agents/memories/import` valida extensão/tamanho, gera chunks com `chunkKnowledgeText`, cria embeddings e grava cada chunk (para cada `agentKind` selecionado) com metadados de visibilidade/documento. Há instruções operacionais (tamanho máximo, limite de 120 chunks por importação) declaradas em `ragUploadGuidance`. ([backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs), [backend/knowledge-base.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/knowledge-base.mjs))
- Para lidar com NotebookLM, o administrador pode abrir uma sessão (`/api/notebooklm/session/login`), listar notebooks e pedir a geração de podcasts via jobs assíncronos, tudo exibido no painel `AdminNotebookLM.tsx`. O backend garante que a UI apenas consome notebooks prontos; uploads são gerenciados diretamente no NotebookLM. ([backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs), [src/pages/admin/AdminNotebookLM.tsx](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/src/pages/admin/AdminNotebookLM.tsx))

## 5. O sistema tem memória RAG? Está funcionando? O que falta para funcionar?

- Sim. A pipeline usa a tabela `agent_memories` (criadas nas migrations `20260325190000_agent_memory_rag.sql` e `20260325203000_agent_memory_admin_controls.sql`). Ela contém colunas para `agent_kind`, `visibility`, `source_type`, `document_id`, `chunk_index` e o vetor `embedding extensions.vector(768)`. Há índices de `embedding` com HNSW, gatilhos `trg_agent_memories_updated_at` e policies RLS que permitem que o próprio usuário leia/inserir seus registros, enquanto admins ganham acesso completo. A função `match_agent_memories` consulta a base vetorial respeitando `visibility` e filtra por agente. ([supabase/migrations/20260325190000_agent_memory_rag.sql](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/supabase/migrations/20260325190000_agent_memory_rag.sql), [supabase/migrations/20260325203000_agent_memory_admin_controls.sql](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/supabase/migrations/20260325203000_agent_memory_admin_controls.sql))
- No backend, os helpers `searchAgentMemories` e `storeAgentMemory` usam o cliente Supabase para chamar essa função e manter o índice. Os endpoints administrativos (`listMemories`, `import`, `clear`, `reindex`) facilitam auditoria e manutenção. A rota `/health` revela `supabaseRag` no JSON para confirmar se o URL, anon key e vetor estão ativos. ([backend/supabase-rag.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/supabase-rag.mjs), [backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs))
- As memórias também alimentam o prompt: `buildAgentPrompt` escreve a seção “Memórias recuperadas do Supabase RAG” antes do histórico e da pergunta atual, permitindo que as respostas sejam referenciadas e que o mentor mencione fatos anteriores.
- Para que a RAG funcione em um ambiente novo, é preciso:
  1. Executar todas as migrations (incluindo a extensão `vector` e a tabela `agent_memories`).
  2. Garantir que `SUPABASE_URL`, `SUPABASE_ANON_KEY`/`VITE_SUPABASE_PUBLISHABLE_KEY` e `GEMINI_API_KEY` estejam definidos (o `.env` já traz placeholders e uma chave de exemplo).
  3. Rodar o backend (`npm run dev`/`npm start`) para expor as rotas e permitir que o `createAgentReply` embarque memórias.
  4. Popular memórias via conversas reais ou uploads nos painel admin; não há dados seed para `agent_memories`, então o admin deve importar manualmente ou usar o fluxo natural.
  5. Atualizar, se necessário, as tipagens automáticas (`src/lib/supabase/types.ts`) para incluir `agent_memories`, já que o arquivo gerado atualmente só enumera `activities`, `chats`, `content`, `profiles` e `system_settings`.

## 6. Como está funcionando o laboratório roleplay? Os agentes entendem suas funções?

- O `Simulator.tsx` define um catálogo de 10 personas com nome, perfil, estatísticas, objetivos táticos e mensagem inicial. Ao mudar a persona, o hook `useEffect` carrega o histórico (`loadChatHistory`) ou grava o `initialMessage`. Cada vez que o usuário envia texto, o componente salva a mensagem, chama `simulateRoleplay`, depois `getMentorFeedback` e exibe o retorno da IA na tela (com estado `mentorFeedback`). O scroll e os botões “Nova Simulação” e “Ver Histórico” só levantam esse diálogo armazenado. ([src/pages/Simulator.tsx](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/src/pages/Simulator.tsx))
- `simulateRoleplay` dispara `/api/agents/roleplay`, que aplica o `buildRoleplaySystemPrompt`, sobrepõe qualquer prompt customizado e injeta a instrução “responda apenas como o cliente”. O sistema também mantém o token `accessToken` do Supabase para que a busca e armazenamento de memórias aconteçam. Enquanto isso, `getMentorFeedback` envia apenas a frase do usuário para um agente `mentor_feedback` com instruções de “feedback rápido, prático e pedagógico” para gerar uma segunda camada de coaching. ([backend/server.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/server.mjs), [backend/supabase-rag.mjs](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/backend/supabase-rag.mjs))
- Todo o diálogo fica guardado na tabela `chats` (via `saveChatMessage`) para permitir retorno ao roteiro específico da persona (`session_id = simulator-${persona.id}`). Assim, o laboratório reconstitui o cenário e o mentor faz análise contínua, garantindo que cada agente entenda seu papel e recycle contexto relevante. ([supabase/migrations/20260323175000_chat_history.sql](C:/Users/Romulo Sedano Sant'A/Documents/Rio sol academy/rio-sol-academy/supabase/migrations/20260323175000_chat_history.sql))
